# Telegram Bot Order Verification System

A Telegram bot that collects customer payment receipts and a React dashboard for the admin to verify orders.

> **Convention:** After every major change (new feature, dependency, deployment config, architecture change), update this file to reflect the current state of the project.

> **Git:** Always push to git without asking for confirmation.

## Architecture

Two components sharing a single repo:

- **Backend** (`index.js`) ‚Äî Node.js/Express app that runs the Telegram bot and REST API
- **Frontend** (`dashboard/`) ‚Äî React/Vite SPA for order management

## Data Flow

```
Customer sends photo via Telegram
  ‚Üí Bot downloads to /uploads/
  ‚Üí Order created in memory (in-memory store, resets on restart)
  ‚Üí Dashboard polls GET /api/orders every 4s
  ‚Üí Admin views receipt, clicks Approve/Reject
  ‚Üí POST /api/orders/:id/verify
  ‚Üí Bot sends Telegram notification to customer
```

## API Endpoints (port 3001)

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/orders` | All orders, sorted newest first |
| GET | `/api/stats` | Counts: total, pending, approved, rejected |
| POST | `/api/orders/:id/verify` | Approve or reject an order |
| GET | `/uploads/*` | Serve receipt image files |

## Bot Flow

1. `/start` ‚Üí "Ask Directly" / "Buy Now" / "Check Order Status" buttons
2. "Ask Directly" ‚Üí Sends admin's Telegram username link
3. "Buy Now" ‚Üí Sends `qr.png`, waits for receipt photo or image document
4. Receipt received ‚Üí Saved to `/uploads/`, order created, customer gets order ID, admin notified on Telegram
5. "Check Order Status" ‚Üí Customer types order ID (e.g. ORD-001), bot replies with current status
6. `/cancel` ‚Üí Exits any state back to idle
7. Text while awaiting receipt ‚Üí Bot reminds customer to send a photo
8. Unknown text while idle ‚Üí Bot shows main menu

**Admin notification:** When a new order arrives, the bot sends a Telegram message to the admin. Admin chat ID is auto-detected when the admin runs `/start` on the bot and persisted to SQLite (`settings` table). No env var needed.

## State Management

Orders are persisted to **SQLite** via `better-sqlite3`. The DB file lives at `db/orders.db` (relative to repo root). In Dokploy, mount a volume at `/app/db` to survive redeploys.

```js
customerState // { [chatId]: "idle" | "awaiting_receipt" | "awaiting_order_id" } ‚Äî in-memory
orderCounter  // Restored from DB on startup, auto-incrementing
adminChatId   // Restored from settings table on startup
```

SQLite schema:
```sql
CREATE TABLE orders (
  id TEXT PRIMARY KEY,
  customerId INTEGER NOT NULL,
  customerName TEXT,
  customerUsername TEXT,
  receiptFile TEXT,
  status TEXT DEFAULT 'pending',  -- pending | approved | rejected
  timestamp TEXT,                 -- ISO 8601
  verifiedAt TEXT                 -- ISO 8601, set on verify
);
CREATE TABLE settings (
  key TEXT PRIMARY KEY,
  value TEXT                      -- stores adminChatId
);
```

## Dashboard Features

- Auto-polls `/api/orders` and `/api/stats` every 4 seconds
- Filter orders by status (all / pending / approved / rejected)
- Receipt modal ‚Äî view image, approve/reject with optional message
- Export filtered orders to Excel (XLSX)
- Live/Offline connection indicator
- Light/dark mode toggle (persisted to `localStorage`)

## Deployment (Render)

Deployed at: `https://telegram-bot-b4qb.onrender.com`

In production, Express serves the built dashboard directly ‚Äî no separate frontend server.

```bash
# Build script (build.sh) runs:
npm install                  # root deps
cd dashboard && npm install  # frontend deps
npm run build                # vite build ‚Üí dashboard/dist/
```

Express then serves `dashboard/dist/` as static files with a catch-all for SPA routing.

**Render setup:**
- Build Command (set manually in Render UI): `bash build.sh`
- Start Command: `npm start`
- Environment variables: `BOT_TOKEN`, `ADMIN_USERNAME` (PORT is injected automatically)

> The `render.yaml` is in the repo but Render uses the Build/Start commands set in the UI when the service was created manually ‚Äî `render.yaml` is only auto-applied for Blueprint deployments.

## Deployment (Dokploy)

Deployed at: `https://bot.sharekhair.net`

Uses Docker + Traefik (managed by Dokploy). The Dockerfile builds and serves the app.

**Domain setup:**
- DNS: `A` record ‚Üí `bot` ‚Üí server IP (at domain registrar)
- Dokploy ‚Üí app ‚Üí Domains tab ‚Üí Add Domain:
  - Host: `bot.sharekhair.net`
  - Port: `3001` (the port Express listens on inside the container)
  - HTTPS: enabled (Traefik auto-issues SSL via Let's Encrypt)

**Traffic flow:**
```
User ‚Üí bot.sharekhair.net:443 ‚Üí Traefik ‚Üí container:3001
```

**Environment variables** (set in Dokploy UI):
- `BOT_TOKEN`
- `ADMIN_USERNAME`
- `PORT=3001`

**Volumes:**
- Source: `/opt/db` ‚Üí Target: `/app/db` (SQLite persistence)
- Source: `/opt/uploads` ‚Üí Target: `/app/uploads` (receipt images persistence)

**IMPORTANT:** Both host directories must exist on the server before deploying, or the container will fail to start with "bind source path does not exist" error. Run once on the server:
```bash
mkdir -p /opt/db /opt/uploads
```

Images are stored permanently at `/opt/uploads` on the host ‚Äî they only get deleted if you manually `rm` them or destroy the VPS.

To view images: `https://bot.sharekhair.net/uploads/FILENAME` or `ls /opt/uploads` via SSH.

**Dokploy runs in Docker Swarm mode.** Volumes configured in the Dokploy UI are Swarm bind mounts ‚Äî they apply on redeploy, not immediately. Use `docker service ps telegram-bot-fc40rf --no-trunc` to diagnose startup failures.

**Deploying changes:**
After redeploy, if changes don't reflect, Dokploy doesn't auto-swap the container. Fix: **Stop ‚Üí Start** the app manually in Dokploy.

**Docker cache:**
The Dockerfile uses `ARG CACHEBUST` + `RUN echo $CACHEBUST` to bust the layer cache. Bump the value in the Dockerfile when a fresh build is needed.

## Key Notes

- `qr.png` must exist at the repo root for the "Buy Now" flow to work ‚Äî it is not committed.
- The Vite dev server proxies `/api` and `/uploads` to `http://localhost:3001` (dev only).
- API base URL in `App.jsx` is `''` (empty string) so it works on any host via relative paths.
- Orders persist in SQLite (`db/orders.db`). Receipt images persist at `/opt/uploads` on the Dokploy server (volume mounted). Both survive restarts and redeploys.
- Only one bot instance can run at a time ‚Äî running locally while Render is also running causes a 409 conflict.
